## 完成項目

### 1. 新建 SQL RPC（admin/JWT 版 2 支）

**Migration 檔**：`docs/sql/migrations/2026-04-29_legacy_admin_orders.sql`

| RPC | 用途 |
|---|---|
| `get_legacy_all_orders_admin(secret, date_from, date_to, store_name, order_type, search, limit)` | 舊單列表（admin 不指定店家可看全部分店）|
| `get_legacy_order_details_admin(secret, order_no)` | 單張舊單明細（modal 用，回 JSONB）|

**安全模式**：admin_secret OR JWT role 二擇一驗證（跟 simple_*_admin 同款）。前端網站只走 JWT 路徑（admin_secret 不放 public repo）。

**過濾**：
- `customer_name LIKE '包子媽-%店' OR = '全民'`（避免樂鎂/央廚等其他客戶誤入）
- `p_store_name` 短名（NULL=全部分店）
- `p_order_type` ('normal'/'return'/NULL=全部)
- 日期區間用 `order_date`（舊單沒 delivery_date）
- `p_search` 對 order_no ILIKE

**回傳格式對齊 simple_get_all_orders**：
- `store_name` 用短名（從 `包子媽-XX店` 萃取 XX），跟 simple_sales_orders 對齊
- `total_amount` 保留原值（RT 是負數，前端用紅字顯示讓 admin 一眼識別）
- 多回 `order_type` / `ref_order_no` 給前端類型欄 + RT「↩️ 原單」徽章用

**型別 cast 注意**：sales_orders 的 `customer_name` 和 `ref_order_id` 都是 `varchar`，不是 `text`。RETURNS TABLE 嚴格檢查型別，CASE 表達式只要有一個分支回 varchar 就會被推論成 varchar → 跟 TEXT 不對盤。所有 sales_orders 字串欄位都用 `::TEXT` cast。

不另外做批次 RPC：舊單暫不支援列印（決策 Q2=B），批次列印 / Excel 匯出在前端 filter 時排除 SO/RT。

## 已知限制 / 設計取捨

1. **舊單 status 一律「已建單」**：admin 篩選「已建單」會混到舊單。已用「📦 一般」/「📜 歷史」徽章區分。如果只想看新系統，用日期區間限制（如 from=2026-04-22）。

2. **舊單沒 delivery_date**：用 order_date 代，列表「訂單日 = 出貨日」會重複顯示。視覺上不漂亮但資料正確。

3. **沒加「資料類型」獨立篩選器**：UI 維持原 5 個篩選器（日期/店家/狀態/退貨/搜尋）。要篩 RT 可用單號搜尋輸入「RT」。

4. **舊單暫不支援列印**：admin 可在 modal 看明細對帳，要實際列印請去舊系統 branch_admin / index.html 查詢列印（決策 Q2=B，避免做新 print 模板）。

5. **舊資料雜訊已過濾**：sales_details.return_status 有 'none' / 'NULL' / 英文值（accepted/requested 等）的舊資料，sidebar 已過濾或翻譯，DB 不動避免影響 branch_admin 報表邏輯。
