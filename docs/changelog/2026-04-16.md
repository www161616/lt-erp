# 2026-04-16 事故與修復紀錄

## 🚨 重大事故：branchOrders 4 月資料全部遺失 + 救援

### 症狀
- admin 開團總表打開後，**所有 21 家分店的 4 月資料全部看不到**
- 點選 4 月任一結單日（04-01 ~ 04-15）→ 空白
- 1、2、3 月歷史資料完整，只有 4 月不見

### 診斷過程
1. F12 Console 腳本檢查 shared_kv.branchOrders 發現：
   - **雲端和本機一模一樣**（重整救不回來，雲端就是壞的）
   - 各店 `_4/X` 格式的 key **全部 0 筆**
   - 各店 `_3/X` 格式的 key 完整（3 月正常）
   - branchOrderList 的 4 月品項完整（667 筆），但 branchOrders 對應數量全失
2. 對照 CLAUDE.md 04/14 紀錄：
   - 經國從 3720 筆掉到 2467 筆（少 1253 筆）
   - 萬華從 1124 筆掉到 1116 筆（少 8 筆）
   - 確認不只是顯示問題，是雲端 key 真的被洗掉

### 救援來源
- **Google Drive 備份**：`LT-ERP 備份` 資料夾 → `2026-04` Sheet → **04-15 分頁**（2026-04-15 02:43:22 快照）
- ⚠️ Google Drive 同資料夾的 **JSON 備份檔有 bug**（見下方「發現的新問題」），完全抓不到 4 月 key
- ⚠️ **今天（04-16）凌晨 2 點應該要跑的備份完全沒產生**，只能用 04-15 那份

### 救援方法
1. 備份現狀（下載當前壞掉的雲端 JSON 當保險）
2. 匯出 04-15 Sheet 分頁為 CSV（含 4 月 445 筆商品 × 21 家店矩陣）
3. 寫獨立 HTML 還原工具：
   - 用 **fetch-before-merge 模式**：每家店還原前先抓雲端當前 data → 合併 4 月 key → 用 `merge_branch_order` RPC 推回
   - 循序執行（不併發），避免互相覆蓋
   - 內建「已有 4 月 key 則跳過」防呆（防重複執行）
4. 先測試 3 家小店（買上癮 / 龍潭 / 萬華）驗證邏輯
5. 批次還原剩餘 17 家店
6. 最終驗證：20 家店 4 月 key 數全數與備份符合

### 救援結果
- 20 家店 4 月 branchOrders 完全還原
- 總共還原 4691 個 key（跨 20 家店）
- 3 月以前歷史資料**零影響**（fetch-before-merge 保證）
- 04-15 凌晨 2 點到 04-16 下午救援這段 1.5 天，portal 未關閉的分店本機 localStorage 還在；已關閉的分店的新增資料無法還原

### 要求店家驗證的事
- 04-15 ~ 04-16 這段時間新填的 4 月訂購數字：
  - portal 沒關過 → 本機還在，會自動 cloudSave 同步
  - portal 關過或重整過 → 資料可能遺失，需要重填

## 🔍 發現的新問題（待追查/待修）

### 1. 🔴 Google Apps Script JSON 備份有 bug（高優先）
- **症狀**：04-15 JSON 備份檔 `lt-erp-backup-2026-04-15.json` 打開後，branchOrders 各店的 `_4/X` key **全部 0 筆**
- **對照**：同一天凌晨 2:43 的 Sheet 備份（04-15 分頁）**有完整 4 月資料**（5570 筆 2026-04 記錄）
- **推論**：JSON 匯出函式和 Sheet 匯出函式抓的資料源不同，或 JSON 匯出有過濾邏輯漏掉 4 月格式
- **風險**：JSON 一直被當作主備份來源（CLAUDE.md 04/14 記載「Google Drive JSON：每日存完整 JSON 檔（可用於還原）」），實際上不可用
- **修法**：檢查 Apps Script 備份專案的 JSON 匯出函式，對比 Sheet 匯出邏輯找出差異

### 2. 🔴 04-16 凌晨 2 點備份沒跑（高優先）
- **症狀**：04-16 當天的自動備份完全沒產生（Sheet 沒新分頁、JSON 沒新檔案）
- **推論**：可能是 Apps Script trigger 被停用、或腳本執行失敗
- **修法**：登入 script.google.com → 備份專案 → 檢查「執行項目」歷史，看失敗原因

### 3. 🔴 branchOrders 4 月整批消失的根因未查明（最高優先）
- **事實**：04-15 凌晨 2 點（Sheet 備份時）branchOrders 4 月還完整。04-16 下午發現時 4 月全失
- **中間 1.5 天內發生什麼**：未知
- **可能的觸發點**：
  - admin 的某個操作觸發 cloudSave 整份覆蓋（但 branchOrders 有 dirty tracking 保護，04/14 才修過）
  - QuotaExceeded 清除（但 branchOrders 不在可清除名單）
  - 某個 bug 繞過了 `merge_branch_order` RPC 直接 POST
  - 員工/助理某次操作的未追蹤副作用
- **調查方向**：
  - Supabase shared_kv 雲端最後更新時間：`2026-04-16T13:34:58`（還原前）
  - 這個時間點前後查所有人的 Line 紀錄 / 螢幕紀錄
  - grep 所有 `shared_kv` 的 POST（非 RPC）呼叫點，檢查是否有繞過 merge_branch_order 的地方
- **如果不查清楚，下次還會再發生**

- 要麼是 RPC 本身有問題
