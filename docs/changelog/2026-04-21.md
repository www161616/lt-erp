## 🧠 根因候選（更新）

有更多資料後，上次的「連續建單污染 RPC」假設更像：

- 04/14：3 張連續 → 全部或多數失敗
- 04/15：3 張連續 → 全部或多數失敗
- 04/16：2 張連續 → 第一張部分失敗 + 第二張全失敗

**假設**：某個全域狀態在「第一次 generateSalesOrder 開始」時被污染，影響後續所有 for-await-of 迭代的 RPC call 的 items 傳值。污染源可能是：
- admin 頁面某個 setInterval 在 async 中改了 wave 物件
- 或是 generateSalesOrder 內部某個 shared state

**下次揀貨分發要盯緊**：
- 如果只跑 1 張揀貨單 → 可能正常
- 如果連續跑 ≥2 張 → 可能觸發

**Debug 時要做的**：在 `generateSalesOrder` 的 for-await-of 前後 + await RPC 前後加 `console.log(JSON.stringify({branch, items_len: items.length, items_qty: items.map(i=>i.qty)}))`，就能看到 items 何時被清空。

## 🔴 下午：連環 bug 發現（完整紀錄）

上午補完 81 張後，使用者回報 portal 端看到銷貨單日期顯示「2026-04-20」而非原本的 04/14~04/16。調查一路展開。

### 第 1 層：order_date 被改成 2026-04-20

**查詢**：
```sql
SELECT id, order_date, created_at, grand_total, wave_id, note
FROM sales_orders WHERE id IN (幾張代表);
```

**結果**：
- SO20260414048 (wave=PICK-697028, created 04-14) → `order_date = 2026-04-20` ❌
- SO20260415031 (wave=PICK-809644, created 04-15) → `order_date = 2026-04-20` ❌
- SO20260416032 (wave=PICK-492631, created 04-16) → `order_date = 2026-04-20` ❌

主表 note 是今天寫的 `[2026-04-21 手動補回 sales_details]`，但 order_date 是昨天 04-20 → 代表**不是今天的 PATCH 改的**，是昨天（2026-04-20）就已經被改壞。

### 第 2 層：全體規模掃描（order_date 被改的）

```sql
SELECT order_date, COUNT(*)
FROM sales_orders
WHERE wave_id IN (8 個 BUG-014 受害 wave)
GROUP BY order_date;
```

結果：
- 2026-04-14: 6（原本成功的正常單）
- 2026-04-15: 9
- 2026-04-16: 8
- **2026-04-20: 102（全部被改成 04-20）**

剛好 102 張 = BUG-014 受害數。

### 第 3 層：payment_status 也被改

使用者截圖：在 index.html 銷貨單列表，補完的單全部顯示「已收款」。

```sql
SELECT wave_id, payment_status, COUNT(*)
FROM sales_orders
WHERE wave_id IN (8 個受害 wave)
GROUP BY wave_id, payment_status;
```

**關鍵**：note = `手動補回` 的 → payment_status = `已收款`；note = null（原本成功）的 → payment_status = `未收款`。

### 第 4 層：誤判是我的 PATCH 造成

一度 100% 歸咎我的 PATCH（因為相關性完美）。但後來查：
- sales_orders.payment_status 的 DB default = `'未收款'`（不是已收款）
- information_schema.triggers 查 sales_orders → **沒有 trigger**
- 04-17 之後新建的正常分店單（用 web_save_sales_order RPC）payment_status = `未收款`

**那為什麼補過的單變已收款？**

### 第 5 層：反證來自 PICK-757876

使用者發現 SO20260402015 / SO20260402026 也是 0 項 $0 + 已收款 + order_date=04-20。查 DB：

```sql
-- 關鍵欄位
id | payment_status | grand_total | order_date | note
SO20260402015 | 已收款 | 0 | 2026-04-20 | null
SO20260402026 | 已收款 | 0 | 2026-04-20 | null
```

**note = null = 我從未 PATCH 過**，但 order_date=04-20、payment_status=已收款 照樣壞了 → 證明**不是我的 PATCH 造成**。

### 第 6 層：真相

**04-20 當天有某個操作（人或系統）對所有 `grand_total = 0` 的分店單做批次 UPDATE**：
- `order_date` → 2026-04-20
- `payment_status` → 已收款

對象包括：
- 04/02 PICK-757876 的 2 張（我擱置的）
- 04/14~04/16 BUG-014 受害的 102 張（我補過的）

剛好 102 張 = BUG-014 受害數**是巧合**。我的 PATCH 只加 note，沒動 order_date / payment_status，但這兩個欄位已經在 04-20 就被改壞了。

**源頭不明**。員工也會操作 admin，可能是員工點過某個按鈕觸發批次 UPDATE。

### 第 7 層：SQL 修復 102 張

測試 SQL UPDATE 單張（驗證沒有副作用）：
```sql
UPDATE sales_orders SET payment_status = '未收款' WHERE id = 'SO20260414014';
SELECT id, payment_status FROM sales_orders WHERE id = 'SO20260414014';
-- 結果：未收款 ✓（SQL UPDATE 不會被 trigger 改回去）
```

批次修 102 張：
```sql
UPDATE sales_orders
SET payment_status = '未收款',
    order_date = TO_DATE(SUBSTRING(id FROM 3 FOR 8), 'YYYYMMDD')
WHERE note LIKE '%手動補回 sales_details%'
  AND id ~ '^SO[0-9]{11}$';
```

驗證：
- payment_status: 102 張全部 `未收款` ✓
- order_date 分布：04-14 (40) + 04-15 (41) + 04-16 (21) = 102 ✓

## 🔴 PATCH sales_orders 的副作用（新 bug）

### 現象
透過 PostgREST REST API 做 PATCH sales_orders（即使 body 裡不包含 payment_status）→ payment_status **會被改成 `已收款`**。

SQL UPDATE（直接 DB）**沒**這個副作用。

### 證據
- 我昨天和今天的 Stage 2 腳本：PATCH body 只含 subtotal / tax_amount / grand_total / note
- 結果所有被 PATCH 過的 102 張 → payment_status = 已收款
- 沒被 PATCH 過的原正常單 → payment_status = 未收款（維持不變）
- 反證：04-20 那次批次 UPDATE **可能也是透過 PostgREST PATCH** 做的（這樣才符合「改 order_date 同時 payment_status 變已收款」的 pattern）

### 機制不明
- information_schema.triggers 查 sales_orders → 空
- sales_orders.payment_status 的 column_default = `'未收款'::character varying`
- 可能是：
  - Supabase 的 RLS policy
  - Edge Function 被 webhook 觸發
  - 未在 information_schema 的底層 rule / policy

### 暫時對策
**不要用 REST API PATCH sales_orders**。要修欄位改用 Supabase SQL Editor 的 SQL UPDATE。

```sql

SELECT * FROM pg_policies WHERE tablename = 'sales_orders';
```
