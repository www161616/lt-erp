## 🔍 調查過程

### 1. 確認 DB 狀態（不是顯示 bug）
- SQL 查 `sales_details WHERE order_id LIKE 'SO20260416%'` → 9 張單在 DB 真的**完全沒有 sales_details**
- SQL 查 `sales_orders` 這 9 張主表：**`grand_total = 0`、`subtotal = 0`、`note = null`**
- 主表 `created_at` 連號（100-200ms 間隔），是同一次 `generateSalesOrder` 迴圈產生
- `created_at = 03:09 UTC` = **台灣時間 04/16 11:09**（白天正常時段建單，不是半夜）

### 2. 確認 matrix 資料完整
- shared_kv 的 `lt_savedWaves` 陣列裡有 PICK-397503
- 9 家店 `branchData[店名].actual` **全部都有實分數量**
- `br_picking_items` DB 表有 543 筆揀貨明細，跟 matrix 一致
- `wave.salesOrderGenerated = true`、`converted = true`

### 3. 排除的可能性
- ❌ 讀取 bug：admin + portal 兩邊都是空的
- ❌ items 有但 price 0：`subtotal` 也是 0，代表 items 陣列本身是空的
- ❌ RPC transaction bug：如果主表寫入成功代表 items 空傳，不是明細寫失敗

### 4. 剩下的合理推論（根因未定）
**同一次 `generateSalesOrder` 呼叫，7 張店成功、9 張店 items 空傳**。
可能：
- 當時 admin 本機的 `wave.matrix` 某些 branch 的 `actual` 讀不到（為 undefined 或 object 無 actual 欄位）
- 讓 `const qty = data.actual ?? (typeof data === 'number' ? data : 0)` 變 0
- 但為什麼 7 張 OK 9 張不 OK？**未解**

## 🧠 根因新線索（PICK-492631 提供的）

PICK-492631 的 12 張全失敗提供**比 PICK-397503 更乾淨的線索**：

1. **篩選邏輯正確**：8 家全 0 店確實沒建單（code: `if (qty > 0) { ... }`）
2. **同一個 for-await-of 迴圈連跑兩張揀貨單**（PICK-397503 → PICK-492631，中間 5 秒）
3. **PICK-492631 12 張全失敗**，跟 PICK-397503 的「7 成功 9 失敗」不同

可能假設（待驗證）：
- 某個事件/操作在 03:09:04 發生，**污染了後續所有 RPC call 的 items**
- 第一張揀貨單的後半段 + 第二張整張都受影響
- 污染源可能是：wave 物件被改、items reference 被改、或某個全域狀態

如果這個假設對，根因在**連續建單時的狀態管理**，不是個別 race condition。
下次揀貨分發如果只跑一張單可能不會重現，跑連續兩張才重現。
