### 🔴 嚴重 bug（14 條 — 會損資料、算錯帳、重複建單、帳務爆掉）

#### BUG-AUDIT-001：SalesOrder.html saveOrder 無防連點
- **檔案/行號**：[SalesOrder.html:1149-1221](../../SalesOrder.html)
- **症狀**：會計按「儲存銷貨單」後，如果網路慢或卡住，連點兩下 → 會建出兩張一樣的銷貨單，帳會重覆
- **根因**：`saveOrder(autoPrint)` 是 async 函式但沒有 `_submitting` guard，按鈕也沒 disabled
- **修法方向**：在函式開頭加 `if (_submitting) return; _submitting = true;`，finally 設 `_submitting = false`，按鈕加 `disabled` 管理（參考 `branch_portal.html submitOrder` BUG-004 已修過的寫法）
- **關聯**：BUG-004 同 pattern，分店端已修，會計端遺漏
- **優先序**：🔴 解凍前必修

#### BUG-AUDIT-002：SalesOrder.html 開單時傳 order_date 觸發 RPC 覆寫
- **檔案/行號**：[SalesOrder.html:1165](../../SalesOrder.html)
- **症狀**：按「開立銷貨單」時，如果是編輯既存單（帶 `editingOrderId`），舊單的訂單日期會被覆寫成當天日期。例如今天是 4/23，編輯一張 4/7 的舊單 → 那張單的日期變 4/23 → 月結會亂
- **根因**：前端傳 `order_date: dateStr` 給 `web_save_sales_order` RPC，RPC 在 UPDATE 路徑（第 119 行）會 `UPDATE sales_orders SET order_date = v_date, ...`
- **修法方向**：雙保險修法
  - A. **RPC 端（主修）**：移除第 119 行的 `order_date = v_date,`，UPDATE 路徑永不改日期
  - B. **前端（次修）**：編輯模式不傳 `order_date`
- **關聯**：**等同 BUG-001**（已知，未修）+ 使用者之前討論的 「Bug A」
- **優先序**：🔴 解凍前必修（凍結的主因之一）

#### BUG-AUDIT-003：SearchReturns editReturnOrder 無防連點
- **檔案/行號**：[SearchReturns.html:360-507](../../SearchReturns.html)
- **症狀**：會計在退貨單列表按「✏️ 編輯」，連點多次會觸發多次 RPC 呼叫，可能重複更新庫存（例如原本應該回補 5 個，變成回補 15 個）
- **根因**：`editReturnOrder(orderId)` 無 `_submitting` guard
- **修法方向**：加 guard + 按鈕 disabled
- **關聯**：BUG-004 同 pattern
- **優先序**：🔴 解凍前必修

#### BUG-AUDIT-004：SearchReturns 編輯流程 PATCH + POST 無事務保護
- **檔案/行號**：[SearchReturns.html:439-475](../../SearchReturns.html)
- **症狀**：會計編輯退貨單時，程式會先 PATCH `inventory`（調整庫存）、再 POST `inventory_logs`（寫異動紀錄）。如果中途失敗（網路斷、RLS 擋住等），**庫存改了但沒有 log**，或**庫存沒改但有假 log**
- **根因**：兩個 await 無原子保護，任一步失敗會留下半壞狀態
- **修法方向**：
  - 短期：用 try-catch，POST log 失敗時 rollback PATCH（難度中）
  - 長期：寫一個 RPC `edit_return_order_inventory` 在 SQL 內做原子操作（難度高但安全）
- **關聯**：新發現
- **優先序**：🔴 解凍前必修

#### BUG-AUDIT-005：SearchReturns 作廢退貨 庫存扣回失敗只 console.warn
- **檔案/行號**：[SearchReturns.html:537-551](../../SearchReturns.html)
- **症狀**：會計按「作廢退貨單」時，程式跑迴圈扣回每個商品的庫存。如果某商品的 PATCH 失敗，只寫 `console.warn` 到瀏覽器主控台、**不提示使用者**、迴圈繼續跑。結果：部分商品庫存沒扣回、使用者以為全成功、資料一團亂
- **根因**：
  ```js
  try {
    // PATCH inventory
  } catch(e) { console.warn('庫存扣回失敗:', d.product_id, e); }  // ← 只警告
  ```
- **修法方向**：
  - 失敗時 break 迴圈
  - 累積失敗清單，結束時用 Swal 列出來
  - 或改寫成 RPC 原子操作
- **關聯**：新發現
- **優先序**：🔴 解凍前必修

#### BUG-AUDIT-006：PurchaseOrder 編輯模式三步操作無事務
- **檔案/行號**：[PurchaseOrder.html:867-893](../../PurchaseOrder.html)
- **症狀**：會計編輯進貨單時，程式做三步：
  1. PATCH `purchase_orders`（改主單）
  2. DELETE `purchase_details`（刪舊明細）
  3. POST `purchase_details`（寫新明細）
  
  **任何一步失敗都會留下半壞狀態**：
  - 第 1 步成功、第 2 步失敗 → 主單金額改了但明細沒變 → 對不上
  - 第 2 步成功、第 3 步失敗 → **明細整個空**（類似 BUG-014 的災難）
- **根因**：無 transaction 保護；新增模式有走 RPC `web_save_purchase_order`，但編輯模式沒走 RPC
- **修法方向**：把編輯模式也改走 RPC（需新寫一支 `web_update_purchase_order`），或至少加回滾邏輯
- **關聯**：新發現
- **優先序**：🔴 解凍前必修

#### BUG-AUDIT-007：CustomerMonthlyReport「部分收款」被算成全未收
- **檔案/行號**：[CustomerMonthlyReport.html:343-348](../../CustomerMonthlyReport.html)
- **症狀**：客戶月結對帳單的「已收總額」「尚欠餘額」卡片數字錯的。如果某張單是「部分收款」（例如應收 $1000，已收 $500），這張單會被**整筆**歸到「未收」，**已收那 $500 不算進已收總額**
- **根因**：
  ```js
  if(o.paymentStatus === '已收款') totalPaid += o.grandTotal;
  else totalUnpaid += o.grandTotal;  // ← 部分收款也進這
  ```
  沒查 `payment_received` 表實際收了多少
- **修法方向**：改成查 `payment_received` 表，用實際 paid 金額（參考 Receivablereport.html:277-284 的做法，但要分頁）
- **關聯**：新發現
- **優先序**：🔴 解凍前必修（會計對帳會錯）

#### BUG-AUDIT-008：MonthlyReport「部分收款」被算成全未收

- **修法方向**：同 BUG-AUDIT-007
- **關聯**：新發現
- **優先序**：🔴 解凍前必修

- **關聯**：新發現
- **優先序**：🔴 解凍前必修

#### BUG-AUDIT-010：Receivablereport payment_received 查詢無分頁
- **檔案/行號**：[Receivablereport.html:277](../../Receivablereport.html)
- **症狀**：**這是最危險的一條**。Receivablereport 查 `payment_received` 表抓所有收款紀錄，但沒有加分頁。Supabase REST API 預設只回 1000 筆 → **超過 1000 筆後，舊的收款紀錄查不到** → 那些已收款的單會變成「未收款」狀態 → 會計會**重複收款**
- **根因**：
  ```js
  const payRes = await fetch(`${SB_URL}/payment_received?select=ref_id,amount`, ...);
  ```
  沒有 `limit=1000&offset=N` 的 while loop（CLAUDE.md 明明有規定要分頁）
- **修法方向**：改用 `fetchAllData()` helper（PurchaseOrder.html:549-566 有現成的寫法），用 while loop + offset 抓完
- **關聯**：新發現。CLAUDE.md 已經三令五申要分頁，這裡漏了
- **優先序**：🔴 **第一優先修**

### BUG-AUDIT-030 🔴：Receivablereport payment_received 查錯欄位（不存在的 ref_id）

- **檔案/行號**：[Receivablereport.html:277](../../Receivablereport.html) + 281 行
- **症狀**：線上 Console 一直噴 400 Bad Request。程式**從來沒有真的抓到任何收款紀錄**
- **根因**：`payment_received` 表**沒有 `ref_id` 欄位**。正確欄位是 `order_id`
  - SQL 備份 [第 275 行](../../docs/sql/龍潭總倉ERP_全系統SQL備份_20260228.txt) `INSERT INTO payment_received (id, order_id, customer_id, payment_date, amount, payment_method, note, created_at)` — 沒有 ref_id
  - [ReceivePaymentList.html:163](../../ReceivePaymentList.html) 註解明寫「對齊你的欄位：payment_date, customer_id, order_id, note」，那頁運作正常
- **程式碼片段**：
  ```js
  const payRes = await fetch(`${SB_URL}/payment_received?select=ref_id,amount`, ...);
  //                                                        ^^^^^^ 這個欄位不存在 → 400
  if (p.ref_id) paymentsMap[p.ref_id] = ...;
  //   ^^^^^^  永遠是 undefined
  ```
- **修法方向**：`ref_id` → `order_id`（兩處都要改）
- **關聯**：新發現（從 Console 錯誤挖出）；影響 BUG-AUDIT-010 的修復順序
- **優先序**：🔴 **第一優先修**（不修 030，010 的分頁改了也是白改）

### ⚠️ BUG-AUDIT-010 真相補充（原敘述不完整）

原第 4 節寫 BUG-AUDIT-010 是「payment_received 查詢無分頁」，但這個描述**不完整**。實際狀況：

### 跨檔系統性問題 E（新增）：應收應付報表長期靠 payment_status 硬撐

- 兩份報表（Receivablereport、PayableReport）**長期都是靠 `payment_status` 欄位的 fallback 邏輯**在算已收/已付

- 修完 BUG-AUDIT-030/031 後，「已收/已付」才會真正以 payment_received/payment_made 表的加總為準

### CROSS_FILE_MAP 警告解讀

[CROSS_FILE_MAP.md:536](../../docs/architecture/CROSS_FILE_MAP.md) 有一句：

**解讀**：這可能是指**歷史舊資料**（早期部分 row 把 customer_id 值錯塞進 order_id 欄位）。但看 SQL 備份第 275 行 `INSERT INTO payment_received (id, order_id, customer_id, ...) VALUES (v_payment_id, v_order_id, v_customer_id, ...)` — order_id 和 customer_id 是**兩個獨立欄位**、存不同值。

**結論**：CROSS_FILE_MAP 的警告可能只對**早期錯資料**有效。新資料正確。但修完 BUG-AUDIT-030 後必須**實際驗證**：
1. 跑應收帳款報表，看舊單的已收狀態
2. 如果某些老客戶的金額還是不對 → 去 Supabase 後台 SQL 查 `payment_received` 的 order_id 欄位有沒有塞到 customer_id 格式（`C00XXX` 而不是 `SO...`）的值
3. 有錯資料就寫 SQL migration 修；沒有的話就 OK
